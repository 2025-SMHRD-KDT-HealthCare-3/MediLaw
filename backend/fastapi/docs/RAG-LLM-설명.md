# MediLaw RAG + LLM 아키텍처 설명 (발표 준비용)

> 대상: 개발자 본인의 이해 + 심사위원 대상 발표.
> 모든 수치는 2026-07 기준 코드·DB(`data/medilaw.db`)에서 실측한 값입니다.

---

## 1. 한 장 요약

MediLaw는 **의료·헬스케어 사업자를 위한 법률 컴플라이언스 서비스**입니다.
병원·헬스케어 스타트업이 "환자 데이터를 이렇게 써도 되나?", "이 광고 전단 문제없나?" 같은 질문에
부딪힐 때, **공식 법령·판례 데이터베이스에서 근거를 찾아 그 근거 안에서만 답하는** 챗봇(`/chat`)과
PDF 문서 위험검토(`/documents/review`), 법령 개정현황 대시보드(`/v1/laws`), 연관 판례 그래프
(`/v1/related-graph`)를 제공합니다. 핵심 차별점은 답변 속 모든 법령·판례 인용을 DB와 대조해
**환각(존재하지 않는 인용)을 차단하는 Citation Firewall**입니다.

- **범위**: 핵심 4법 — 의료법 · 개인정보 보호법 · 생명윤리법 · 정보통신망법 (+ 각 시행령·시행규칙)으로 하드필터.
- **정직한 역할 구분**: 답변 문장을 생성하는 LLM 자체는 OpenAI 모델(gpt-5.5)입니다.
  우리의 기여는 **① 법률 특화 하이브리드 검색(RAG), ② 인용 검증(Citation Firewall), ③ 도메인 라우팅·PDF 파이프라인 설계**입니다.

## 2. 전체 흐름도

> **RAG**(Retrieval-Augmented Generation): LLM에게 "이 자료 안에서만 답해"라고
> 검색된 근거를 함께 주고 답을 만들게 하는 기법. 오픈북 시험에 비유할 수 있습니다.

```
사용자 질문
   │
   ▼
┌─────────────────────────────┐
│ ① 도메인 라우터 (3단)         │  범위 밖 질문(부동산·세무 등)은 여기서 거절
│  규칙 → 제약 LLM → 되묻기     │  app/domain_router.py
└─────────────┬───────────────┘
              ▼
┌─────────────────────────────┐
│ ② 하이브리드 검색 (RAG)       │  FTS5 BM25(키워드) ∥ 벡터(의미) → RRF 융합
│  + 동의어 확장, 핵심4법 필터   │  app/rag.py
└─────────────┬───────────────┘
              ▼ 근거 top-k (조문·판례·해석례·결정문·가이드라인)
┌─────────────────────────────┐
│ ③ 답변 생성 (OpenAI gpt-5.5) │  "[근거]에 있는 내용만, [1][2] 번호 인용 강제"
│  SSE 토큰 스트리밍 지원        │  app/llm.py, app/routers/chat.py
└─────────────┬───────────────┘
              ▼ 답변 초안
┌─────────────────────────────┐
│ ④ Citation Firewall          │  답변 속 인용을 정규식 추출 → DB 대조
│  실재·조문·항·시행시점·내용 검증 │  app/citations.py
└─────────────┬───────────────┘
              ▼
답변 + 근거 목록 + 인용검증 결과(확인/주의/오류) + 코퍼스 밖 인용 경고
```

**전부 공식 출처**입니다. 크롤링한 블로그·위키가 아니라, 법제처 국가법령정보 Open API와
보건복지부 공식 게시판에서 수집해 SQLite(`data/medilaw.db`) 한 파일에 담았습니다.

| 데이터 | 건수 (실측) | 출처 |

| 판례 (cases) | **20,994건** | 법제처 국가법령정보 Open API |
| 법령해석례 | **93건** | 법제처 국가법령정보 Open API |
| 개인정보보호위원회 결정문 | **344건** | 법제처 국가법령정보 Open API |
| 보건복지부 가이드라인 | **28건** | 복지부 공식 게시판 |

  (조례·행정규칙이 핵심 법령을 밀어내는 문제를 실측으로 확인하고 차단).
- 법령 개정 이력(law_revisions 197건)도 배치로 동기화해 "구법 인용" 검증에 사용합니다.

## 4. 검색(RAG): 어떻게 정확한 근거를 찾나

한 가지 검색만으로는 법률 질문을 못 잡습니다. 그래서 **두 검색을 병행하고 융합**합니다.

- **키워드 검색 (FTS5 BM25)**: "단어가 정확히 들어있는 문서"를 찾음 — 도서관 색인 카드 방식.
  조문번호·법령명처럼 정확한 표현에 강함.
  > BM25: 검색어가 문서에 얼마나 자주·희소하게 등장하는지로 점수를 매기는 고전 검색 알고리즘.
- **벡터 검색 (의미 검색)**: 문장을 숫자 좌표(임베딩)로 바꿔 "뜻이 비슷한 문서"를 찾음 — 사서에게
  말로 설명하면 알아듣는 방식. 표현이 달라도 의미가 같으면 잡아냄.
  > 임베딩: OpenAI `text-embedding-3-small`, **512차원**, sqlite-vec 확장으로 SQLite 안에서 코사인 검색.
- **RRF 융합** (Reciprocal Rank Fusion): 두 검색의 "순위"만 보고 `점수 = Σ 1/(K + rank)`로 합산.
  점수 체계가 다른 두 검색을 공정하게 합치는 표준 기법입니다.

**법률 특화 처리 두 가지** (우리가 직접 설계한 부분):

1. **법률용어 동의어 확장** — 일상어와 법조문 표제어의 불일치를 보정합니다.
   예: 사용자가 "CCTV"라고 물으면, 개인정보 보호법 제25조는 "**고정형 영상정보처리기기**"라고
   쓰기 때문에 키워드 검색이 실패합니다. 검색 전에 "CCTV → 고정형 영상정보처리기기 영상정보"로
   질의를 확장해 해결했습니다(`app/rag.py`의 `_FTS_SYNONYMS`).
2. **FTS 단계 핵심법령 필터** — 이름이 비슷한 행정규칙 복제 조문 수십 건이 진짜 법조문의
   RRF 점수를 희석시키는 문제를 실측하고, BM25 단계부터 핵심 법령만 올리도록 보정했습니다.

멀티턴 대화에서는 "그럼 처벌은?" 같은 후속질문을 LLM으로 **독립 질의로 재작성**한 뒤 검색합니다.

## 5. 생성(LLM): 어떻게 답변을 만드나

생성 모델은 OpenAI **gpt-5.5** (`app/config.py`의 `CHAT_MODEL`, 환경변수로 교체 가능)이고,
`/chat/stream`에서 SSE(Server-Sent Events) 토큰 스트리밍을 지원합니다.
핵심은 모델이 아니라 **모델을 가두는 방식**입니다.

**근거 안에서만 답하게 하는 프롬프트 규칙** (`app/routers/chat.py`):

1. "[근거]에 있는 내용만 사용해 답하라."
2. "근거에 없으면 추측하지 말고 '제공된 자료로는 확인되지 않습니다'라고 답하라."
3. "사용한 근거는 문장 끝에 [1], [2] 번호로 인용하라." → 프론트에서 근거 카드와 연결
4. "법령명·조문번호는 근거에 적힌 그대로 쓰라. 없는 조문을 만들지 마라."
5. "[근거]에 없는 다른 법령·판례는 이름조차 언급하지 마라(약사법·형법 등 4법 밖)."

**3단 도메인 라우터** (`app/domain_router.py`) — 범위 밖 질문은 답하지 않습니다:

| 단계 | 방법 | 비용 |
|---|---|---|
| 1차 | **결정론 키워드 규칙** — 의료 키워드("환자","진료"...) 있으면 즉시 답변 확정, 완전 무관하면 즉시 거절 | LLM 호출 0회 |
| 2차 | 모호한 경우만 **제약 LLM 분류기** 1회 (JSON 한 줄만 출력하게 강제) | LLM 1회 |
| 3차 | 분류 실패·저신뢰면 **되묻기**(needs_clarification) — 억지로 답하지 않음 | — |

예: "야근수당 계산법?" → 즉시 거절. "사무실 CCTV 안내문 붙여야 하나요?" → 의료기관인지 되묻기.

## 6. 신뢰성: 환각을 어떻게 막나 — Citation Firewall (심사 핵심 어필 포인트)

> 환각(hallucination): LLM이 존재하지 않는 법조문·판례를 그럴듯하게 지어내는 현상.
> 법률 서비스에서는 단 한 건도 치명적입니다.

프롬프트로 "지어내지 마"라고 하는 것만으로는 부족합니다. MediLaw는 **답변이 나온 뒤에
모든 인용을 기계적으로 재검증**합니다 (`app/citations.py`, 612줄).

**동작 단계**:

1. **추출** — 답변 텍스트에서 "의료법 제27조 제1항", "대법원 2013도850" 같은 법령·판례
   인용을 정규식으로 전부 뽑아냅니다.
2. **실재 검증** — DB와 대조: 그 법령이 존재하는가? 그 **조문**이 존재하는가? 그 **항(①②③)**이
   조문 본문에 실제로 있는가? 인용한 시점(as_of)에 **시행 중**이었는가?
3. **내용 일치 검증** — 구조가 맞아도 내용을 왜곡했을 수 있으므로, 답변의 주장 문장과 실제 조문
   본문을 임베딩(text-embedding-3-large, 3072차원)으로 비교해 의미가 일치하는지 확인합니다.
4. **구법 리스크 교차검증** — law_revisions 개정 이력과 대조해, 시행예정 개정이 있는 조문이면
   "곧 바뀔 수 있음"을 경고합니다.
5. **등급화** — 인용마다 `확인 / 주의 / 오류` 상태와 신뢰점수를 매겨 답변과 함께 반환합니다.
   조문·항이 실재하지 않으면 **오류(환각)**로 표시됩니다.
6. **코퍼스 밖 인용 경고** — 4법 밖 법령을 인용하면 "환각"으로 단정하지 않되(실재할 수 있으므로)
   "검증 불가" 경고줄을 답변 끝에 명시해 사용자가 오인하지 않게 합니다.

**한 줄 요약: "답변에 등장하는 모든 법령·판례 인용은 DB와 대조되어 실재 여부가 확인됩니다."**
이것이 일반 챗봇과 구조적으로 다른 지점입니다.

## 8. 보조 파이프라인 요약

### PDF 문서 위험검토 (`/documents/review`, `app/pdf/`)

```
PDF 업로드 → 라우팅(디지털/스캔 판별)
  ├─ 디지털: pdfplumber 텍스트 추출
  └─ 스캔:   OCR — PaddleOCR-VL(자체 호스팅, 데이터 국내처리) 1차
             → 실패 페이지만 OpenAI 비전 폴백, 페이지 4개 병렬
→ 문서유형 분류 → 문단 세그먼트 분할
→ 세그먼트별 RAG 근거 수집 (전 세그먼트 배치 임베딩 1회 + 검색 병렬 실행)
→ LLM 위험판정: 세그먼트 전체를 한 프롬프트에 묶어 JSON 1회 호출 (지연·비용 방어)
→ 원문 최소수정 치환안 생성 → Citation Firewall로 수정안 인용 재검증
```

### 법령 개정현황 (`/v1/laws`)

`scripts/sync_revisions.py` **1일 1회 배치**(cron)로 법제처 데이터를 동기화하고,
미동기화 상태면 첫 호출에서 자기 부트스트랩(24시간 신선도 체크) + 조문 버전 lazy 캐시로
개정 전후 조문 비교표(diff)까지 제공합니다.

### 영어 지원

법령은 법제처 **공식 영문법령**(articles_en, 910건)을 인용하고 "official English"로 표시,
판례·가이드라인은 공식 영문이 없어 **비공식 번역임을 명시**합니다. 영어 질의는 검색 전
한국어로 번역합니다(FTS 인덱스가 한글 토큰이므로).

## 9. 심사위원 예상 질문 Top 7 + 모범 답변

**Q1. 그냥 ChatGPT랑 뭐가 다른가요?**
> 세 가지입니다. ① ChatGPT는 근거 없이 기억으로 답하지만, 저희는 법제처 공식 데이터 21만 건에서
> 근거를 검색해 그 안에서만 답합니다. ② 답변의 모든 인용을 DB와 대조해 환각을 차단하는
> Citation Firewall이 있습니다 — ChatGPT는 존재하지 않는 조문을 확신 있게 말할 수 있습니다.
> ③ 범위 밖 질문은 정직하게 거절합니다. 생성 모델 자체는 OpenAI를 쓰지만, 검색·검증·파이프라인이
> 저희의 기여입니다.

**Q2. 환각(거짓 인용)은 어떻게 막나요?**
> 2중 방어입니다. 생성 전에는 검색된 근거만 주고 "근거에 없으면 모른다고 답하라"고 강제하고,
> 생성 후에는 답변 속 인용을 전부 추출해 법령 존재 → 조문 존재 → 항 존재 → 시행시점 →
> 내용 일치(임베딩 비교)까지 DB로 검증합니다. 통과 못 하면 '오류'로 표시되고, 코퍼스 밖 인용은
> 경고줄이 붙습니다.

**Q3. 데이터는 어디서 가져왔고 저작권 문제는 없나요?**
> 전부 공공데이터입니다. 법령·판례·해석례·결정문은 법제처 국가법령정보 공동활용 Open API,
> 가이드라인은 보건복지부 공식 게시판입니다. 법령·판례는 저작권법상 보호대상이 아닌 공공저작물이고,
> 각 근거에 원문 출처 URL을 함께 제공합니다.

**Q4. 법이 개정되면 어떻게 되나요?**
> 매일 1회 배치(`sync_revisions.py`)로 법제처에서 개정 이력을 동기화합니다. 개정현황 대시보드에서
> 시행예정·현행·연혁을 보여주고, 답변 검증 단계에서도 개정 이력과 교차검증해 "구법 인용 리스크"를
> 경고합니다. 미동기화 상태라도 첫 호출에서 자동 부트스트랩됩니다.

**Q5. 왜 4개 법만 다루나요?**
> 넓고 얕게보다 좁고 검증 가능하게를 택했습니다. 의료·헬스케어 컴플라이언스의 핵심 리스크는
> 의료법·개인정보 보호법·생명윤리법·정보통신망법에 집중되어 있고, 범위를 좁혀야 인용 검증과
> 품질 측정(골든셋)이 실질적으로 가능합니다. 범위 밖 인용은 "검증 불가"로 정직하게 표시합니다.

**Q6. 정확도는 어떻게 검증했나요?**
> 실제 질문 28개와 정답 근거로 골든셋을 만들고 검색 품질을 회귀 측정합니다 — hit@8 100%,
> hit@3 82.1%. 즉 상위 8개 근거 안에 정답이 항상 포함됩니다. 그 외 자동화 테스트 59개로
> 라우터·인용검증·PDF 파이프라인의 동작을 고정하고, 로직 변경 시마다 재측정해 후퇴를 잡습니다.

**Q7. 다른 분야로 확장 가능한가요?**
> 네, 구조가 도메인 독립적입니다. 확장에 필요한 것은 ① 해당 분야 법령·판례 적재(법제처 API 재사용),
> ② 도메인 라우터 키워드 사전, ③ 동의어 사전, ④ 골든셋뿐입니다. 검색·융합·Citation Firewall·PDF
> 파이프라인은 그대로 재사용됩니다. 예: 노동법 컴플라이언스, 금융 규제 검토.

## 10. 시연 추천 시나리오 3개

**시나리오 1 — 동의어 확장 + 근거 인용 (챗봇)**
질문: *"병원 대기실에 CCTV를 설치하려면 어떻게 해야 하나요?"*
보여줄 것: "CCTV"가 법조문 용어 "고정형 영상정보처리기기"(개인정보 보호법 제25조)로 확장되어
정확한 조문·가이드라인이 근거 카드로 뜨고, 답변 문장마다 [1][2] 인용 번호가 달리는 모습.
인용검증 결과가 전부 '확인'인 것까지 보여주면 완결.

**시나리오 2 — PDF 광고 전단 위험검토**
입력: 의원 홍보전단 PDF(저장소에 샘플 `OO의원_홍보전단.pdf` 포함).
보여줄 것: 문단별로 의료광고 위반 위험(예: 치료효과 보장 문구)이 근거 조문과 함께 표시되고,
문서 전체를 다시 쓰는 게 아니라 **문제 문구만 최소 수정**된 치환안이 나오는 모습.

**시나리오 3 — 환각 차단 (Citation Firewall)**
질문: *"의료법 제999조에 따르면 원격의료가 전면 허용된다던데 맞나요?"* 처럼 존재하지 않는
조문을 유도. 보여줄 것: 모델이 근거에 없다며 답변을 삼가거나, 인용이 나오더라도 검증 단계에서
'오류(해당 조문 없음)'로 표시되는 모습 — "저희는 답변을 믿으라고 하지 않고, 검증해서 보여줍니다."
